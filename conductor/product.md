# Product Definition - Elite Dangerous NEXT

## 🎯 Product Vision

**Elite Dangerous NEXT** — это продвинутый анализатор журнальных файлов Elite Dangerous с real-time отслеживанием, статистикой и веб-интерфейсом для командеров (пилотов), желающих отслеживать и анализировать свою игровую активность.

### Value Proposition

| Для кого | Какую проблему решает | Какую ценность создает |
|----------|----------------------|------------------------|
| **Командеры Elite Dangerous** | Отсутствие удобного инструмента для отслеживания игровой статистики и истории действий | Автоматический мониторинг, визуализация и анализ всех игровых событий в реальном времени |
| **Исследователи** | Сложность отслеживания посещенных систем и открытий | Статистика по уникальным системам, категоризация событий исследования |
| **Трейдеры** | Необходимость ручного подсчета торговых операций | Автоматический учет всех торговых операций с категоризацией |
| **Бойцы** | Отсутствие детальной статистики боевой активности | Подсчет боевых событий, отслеживание наград и бонусов |

## 🎮 Domain Context

### Что такое Elite Dangerous?

Elite Dangerous — это космический симулятор, где игроки управляют космическими кораблями, исследуют галактику, торгуют, сражаются и выполняют миссии.

### Что такое Journal Files?

Игра автоматически записывает все действия игрока в журнальные файлы (JSON Lines формат) в папке:
```
%USERPROFILE%\Saved Games\Frontier Developments\Elite Dangerous\Journal.*.log
```

Каждая строка — это JSON-объект с событием (например, `FSDJump`, `Docked`, `Scan`, `MarketBuy`).

### Примеры событий:

```json
{"timestamp":"2026-02-28T14:30:00Z", "event":"FSDJump", "StarSystem":"Sol", "BodyName":"Sol A"}
{"timestamp":"2026-02-28T14:35:00Z", "event":"Docked", "StationName":"Orbis Station"}
{"timestamp":"2026-02-28T14:40:00Z", "event":"Scan", "BodyName":"Earth", "BodyType":"Planet"}
```

## 👥 User Personas

### Primary Persona: Active Commander

**Алексей, 32 года**
- Играет в Elite Dangerous 3 года
- Основной стиль: исследователь/трейдер
- Проводит в игре 10-15 часов в неделю
- **Потребности:**
  - Видеть статистику своих перемещений
  - Отслеживать прогресс исследования галактики
  - Анализировать торговые маршруты
- **Фрустрации:**
  - Нет удобного инструмента для просмотра истории
  - Статистика разрознена по разным источникам

### Secondary Persona: Data-Driven Player

**Мария, 28 лет**
- Играет 2 года, увлекается анализом данных
- Ведет детальные записи своей активности
- **Потребности:**
  - Экспорт данных для внешнего анализа
  - Детальная фильтрация событий
  - Визуализация статистики
- **Фрустрации:**
  - Ручной сбор данных из журналов
  - Отсутствие структурированного хранения

## 📊 User Stories

### Epic 1: Мониторинг журналов в реальном времени

#### US-1.1: Автоматическое обнаружение журналов
> **Как** командер,  
> **Я хочу**, чтобы приложение автоматически находило и читало журнальные файлы Elite Dangerous,  
> **Чтобы** мне не нужно было вручную указывать путь к файлам.

**Критерии приемки:**
- [ ] Приложение автоматически определяет путь к папке журналов на Windows
- [ ] Поддерживается формат `Journal.*.log`
- [ ] Чтение начинается с newest файлов (быстрый старт)
- [ ] Обработка некорректных JSON-строк без падения

#### US-1.2: Real-time отслеживание новых событий
> **Как** командер,  
> **Я хочу**, чтобы новые события из журнала мгновенно отображались в приложении,  
> **Чтобы** видеть свою активность в реальном времени.

**Критерии приемки:**
- [ ] Опрос файла журнала каждые 1 секунду
- [ ] Новые события появляются в UI без перезагрузки
- [ ] WebSocket-уведомления о новых событиях
- [ ] Обработка изменений в файле (дописывание строк)

#### US-1.3: Прогресс загрузки исторических данных
> **Как** командер,  
> **Я хочу** видеть прогресс загрузки исторических журналов,  
> **Чтобы** понимать, сколько времени осталось до готовности.

**Критерии приемки:**
- [ ] Отображение количества загруженных файлов
- [ ] Процент выполнения загрузки
- [ ] Название текущего обрабатываемого файла
- [ ] WebSocket-событие `journal:progress` для UI

---

### Epic 2: Хранение и управление данными

#### US-2.1: Надежное хранение событий
> **Как** командер,  
> **Я хочу**, чтобы все события сохранялись в надежном хранилище,  
> **Чтобы** не потерять данные после перезапуска приложения.

**Критерии приемки:**
- [ ] Использование SQLite (better-sqlite3) для хранения
- [ ] WAL mode для производительности
- [ ] Атомарная запись на диск
- [ ] Сохранение при закрытии приложения

#### US-2.2: Детерминированная идентификация событий
> **Как** разработчик,  
> **Я хочу**, чтобы каждое событие имело уникальный детерминированный ID,  
> **Чтобы** избежать дубликатов при повторной загрузке.

**Критерии приемки:**
- [ ] Формат ID: `{timestamp}_{event}_{uniqueKey}`
- [ ] Извлечение уникального ключа из JSON (StarSystem, BodyName, и т.д.)
- [ ] INSERT OR REPLACE для идемпотентности
- [ ] 100% уникальность event_id

#### US-2.3: Инкрементальная загрузка
> **Как** командер,  
> **Я хочу**, чтобы уже обработанные файлы не загружались повторно,  
> **Чтобы** ускорить запуск приложения.

**Критерии приемки:**
- [ ] Отслеживание обработанных файлов (имя, размер, mtime)
- [ ] Пропуск неизмененных файлов
- [ ] Повторная загрузка при изменении файла

---

### Epic 3: REST API

#### US-3.1: Получение событий с пагинацией
> **Как** разработчик фронтенда,  
> **Я хочу** получать события с поддержкой cursor-based пагинации,  
> **Чтобы** реализовать эффективный infinite scroll.

**Критерии приемки:**
- [ ] `GET /api/v1/events?limit=50&cursor=2026-02-28T14:30:00Z`
- [ ] Возврат формата: `{data, nextCursor, hasMore, total}`
- [ ] Сортировка по timestamp DESC
- [ ] Limit по умолчанию: 50, максимум: 5000

#### US-3.2: Получение статистики
> **Как** разработчик UI,  
> **Я хочу** получать агрегированную статистику по событиям,  
> **Чтобы** отображать dashboard.

**Критерии приемки:**
- [ ] `GET /api/v1/stats`
- [ ] Возврат: `{totalEvents, eventsByType, uniqueSystems, firstEvent, lastEvent}`
- [ ] Группировка по типам событий
- [ ] Подсчет уникальных систем

#### US-3.3: Health check и status
> **Как** оператор,  
> **Я хочу** проверять работоспособность сервера,  
> **Чтобы** мониторить доступность сервиса.

**Критерии приемки:**
- [ ] `GET /health` → `{status: "healthy", version: "1.0.0-beta"}`
- [ ] `GET /api/v1/status` → информация о сервере и БД
- [ ] Время ответа < 100ms

### Epic 4: WebSocket уведомления

#### US-4.1: Real-time уведомления о новых событиях
> **Как** пользователь,  
> **Я хочу** получать мгновенные уведомления о новых событиях,  
> **Чтобы** видеть активность без обновления страницы.

**Критерии приемки:**
- [ ] Socket.IO подключение на `/socket.io/`
- [ ] Событие `new-event` с данными события
- [ ] Отправка при каждом новом событии из журнала
- [ ] Поддержка множественных клиентов

#### US-4.2: Обновление статистики
> **Как** пользователь,  
> **Я хочу** получать обновления статистики,  
> **Чтобы** видеть актуальные данные.

**Критерии приемки:**
- [ ] Событие `stats-update` каждые 5 секунд
- [ ] Данные статистики в payload
- [ ] Автоматическое обновление UI

#### US-5.1: Отображение событий
> **Как** командер,  
> **Я хочу** видеть список событий с детальной информацией,

**Критерии приемки:**

> **Как** командер,  
> **Я хочу** видеть сводную статистику,  
> **Чтобы** оценивать свой прогресс.

**Критерии приемки:**
- [ ] Общее количество событий
- [ ] Разбивка по категориям (прыжки, бой, торговля, исследование)

**Критерии приемки:**

---

### Epic 6: Надежность и отказоустойчивость

#### US-6.1: Graceful shutdown
> **Как** оператор,  
> **Я хочу**, чтобы приложение корректно завершало работу,  
> **Чтобы** не потерять данные при остановке.

**Критерии приемки:**
- [ ] Обработка сигналов SIGINT, SIGTERM
- [ ] Остановка journal watcher
- [ ] Запись данных на диск перед выходом
- [ ] Закрытие всех соединений
- [ ] Timeout 5 секунд для принудительного закрытия

#### US-6.2: Логирование
> **Как** разработчик,  
> **Я хочу** иметь детальное логирование,  
> **Чтобы** диагностировать проблемы.

**Критерии приемки:**
- [ ] 4 уровня логов: ERROR, WARN, INFO, DEBUG
- [ ] Rotation: 5 файлов × 10MB
- [ ] Формат: `[ISO_TIMESTAMP] [LEVEL] [MODULE] message`
- [ ] Изоляция тестовых и продакшен логов

### Core Entities

```
┌─────────────────────────────────────────────────────────┐
│                     EliteEvent                          │
├─────────────────────────────────────────────────────────┤
│ event_id: string (PK, unique)                          │
```
